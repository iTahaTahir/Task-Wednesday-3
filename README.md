# Northwind SQL Analytics

SQL scripts for setting up a Northwind-style sales database and running advanced analytical queries (window functions, CTEs, and conditional aggregation) to answer business questions about employees, suppliers, customers, and products.

## Contents

| File | Description |
|---|---|
| `ddl.sql` | Schema definition (DDL) and full sample data load (DML) for the Northwind database: Categories, Customers, Employees, Shippers, Suppliers, Products, Orders, and OrderDetails. |
| `Analytical.sql` | Five analytical queries answering specific business questions using the schema above. |

## Requirements

- A PostgreSQL-compatible database (the DDL uses `GENERATED BY DEFAULT AS IDENTITY` and `FILTER (WHERE ...)` syntax, both PostgreSQL features)
- A SQL client (e.g., `psql`, DBeaver, pgAdmin, or any tool that can run `.sql` files against Postgres)

## Database Schema

The database models a simplified order-management system:

- **Categories** — product categories (Beverages, Condiments, etc.)
- **Customers** — customer contact and location info
- **Employees** — sales staff details
- **Shippers** — shipping companies
- **Suppliers** — product suppliers
- **Products** — items for sale, linked to a category and supplier
- **Orders** — customer orders, linked to an employee and shipper
- **OrderDetails** — line items per order (product + quantity)

Foreign keys link Products → Categories/Suppliers, and Orders → Employees/Customers/Shippers, and OrderDetails → Orders/Products.

## Setup

1. Create a database (e.g., `Northwind`).
2. Run the DDL script to create tables and load sample data:

```bash
psql -d Northwind -f ddl.sql
```

3. Run the analytical queries:

```bash
psql -d Northwind -f Analytical.sql
```

## Analytical Queries

`Analytical.sql` contains five standalone business questions, each with commentary explaining the insight it provides:

1. **Employee career trajectory** — Identifies employees who handled more orders in the *second half* of their career (by order-date span) than the first half, using a midpoint-based CTE comparison. Useful for spotting employees who ramped up over time (the query currently filters for `orders_first_half < orders_second_half`, i.e., employees who *improved*).

2. **Supplier concentration risk** — For each product category, calculates what percentage of category revenue comes from its single top-grossing supplier, flagging categories as `RISK` if any one supplier accounts for over 60% of revenue. Uses window functions (`SUM() OVER`, `RANK() OVER`).

3. **Customer ordering trend** — For customers with 3+ orders, compares the average gap between orders in the first half of their order history vs. the second half, classifying each customer as ordering more frequently over time or "drifting away" (churn risk). Uses `LEAD()`, `ROW_NUMBER()`, and nested subqueries.

4. **Overpriced, under-performing products** — Finds products in the top 25% by price but in the middle 50% (25th–75th percentile) of total quantity sold — i.e., expensive items that aren't selling proportionally well. Uses `NTILE(100)` for percentile bucketing.

5. **Monthly employee workload & shipping reliability** — Breaks down each employee's order volume by month and splits it into shipped vs. not-shipped counts, using `FILTER (WHERE ...)` conditional aggregation and a `LEFT JOIN` to shippers.

## Usage

Run the whole file to print results for all five questions, or copy individual query blocks (each is self-contained with its own CTEs) into your SQL client to explore one question at a time.

## Notes

- **Known issue in Question 2**: the query joins `OrderDetails` to `Products` using `od.productid = p.productidw` — this references a non-existent column (`productidw`) and will throw an error. It should be `od.productid = p.productid`. Fix this typo before running.
- `Analytical.sql` assumes the schema and data from `ddl.sql` have already been loaded into the same database/session.
- Some queries (e.g., Question 1, Question 3) use integer division and date arithmetic that assume PostgreSQL's date-subtraction-returns-days behavior; behavior may differ on other database engines.
- The sample dataset is the classic "Northwind" demo dataset (customers, orders, etc. from the 1996–1997 period), commonly used for SQL practice and tutorials.
